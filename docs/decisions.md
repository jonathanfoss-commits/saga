# SAGA – selvstendige beslutninger under migreringen

| # | Beslutning | Begrunnelse | Revurderes |
|---|---|---|---|
| D1 | **AEIS beholdes som egen modul** (`board/` + `core/aeis.js`), ikke slått inn i assistant | AEIS er et selvstendig produkt med eget navn, egen arkitektur og fersk videreutvikling på main. Prompten nevner to moduler; AEIS er en tredje med samme rett til modulstatus | Hvis AEIS og assistant vokser sammen |
| D2 | **localStorage-nøkler beholder legacy-navn som datakontrakt** (`jarvis_*`, `aeis_*`, `cf_*`); kun identitetsnøklene får kanonisk `saga_*`-alias med fallback og dobbel-skriv | Eieren har nettopp konfigurert PAT, synk og API-nøkkel i produksjon. Å rename datanøkler gir null brukerverdi og reell risiko for datatap. «Env-variabler» i prompten mappes til disse – aliaset ER renamen, kontrakten er bakoverkompatibiliteten | Når synk-formatet versjoneres (da kan nøkler migreres trygt i samme operasjon) |
| D3 | **iOS-prosjektet røres ikke i kode** | Rename av Xcode-prosjekt/mapper via tekstredigering av pbxproj er velkjent kilde til korrupte prosjekter. Manuell Xcode-rename står i MIGRATION.md | Ved neste iOS-arbeidsøkt |
| D4 | **LLM-klientene i aeis.js og factory.js konsolideres IKKE nå** | AEIS fikk nettopp modellrouting og prompt-caching på main; factory har kostnadsmåler-kontekst. Å tvinge dem sammen midt i en strukturmigrering dobler risikoen i én commit. Ført opp i improvements.md som prioritert kandidat | `saga improve`-loopen |
| D5 | **`/tests/<slug>/` (publiserte falske dører) flyttes ikke** | Live-URL-er skal aldri brekke. Fabrikkens Publish-path er uavhengig av modulstrukturen | – |
| D6 | **Redirect-stubs med JS + meta-refresh** på `/`, `/aeis/`, `/factory/`, pluss selvdestruerende service worker på rot | Installerte PWA-er og bokmerker peker på gamle paths; SW-en på rot cacher det gamle skallet og må aktivt avvikles | Stubs kan fjernes om ~6 mnd |
| D7 | **`saga_activity` blir felles aktivitetslogg; factorys `cf_activity` beholdes og speiles** | Fabrikkens revisjonsspor er del av dens eksport/synk-format; å flytte det ville brutt synkede data. Speiling gir felles logg uten migrering | Ved synk-versjonering |
| D8 | **Tiffany-tekstfarge:** `#0ABAB5` (kontrast ~2,5:1) og `#089490` (~3,7:1) består ikke WCAG AA for liten tekst på hvit. Avledet token `--saga-primary-text: #067a76` (~4,6:1) brukes for små tekstelementer; `#0ABAB5` beholdes for logo, flater og store elementer – som paletten selv foreskriver | Målt med WCAG-relativ luminans | Ved designrevisjon |
| D9 | **Fabrikkens genererte kundeartefakter (nettsteder/apper) får IKKE Tiffany-paletten** | De bærer hvert selskaps egen merkevare (brand-tokens per prosjekt). SAGA-paletten gjelder systemets eget grensesnitt | – |
| D10 | **`saga improve` implementeres som Node-CLI med ærlige grenser** | Stacken har ingen server. Loopen prioriterer og forbereder; implementering av ikke-trivielle endringer delegeres til Claude Code-kall når CLI er tilgjengelig, ellers legges en prioritert plan i changelog-auto.md. Guardrails håndheves i skriptet | Når en kjørende agent-runtime finnes |
| D11 | **n8n/Vercel/databaser/cron finnes ikke i stacken** | Promptens krav om alias for disse mappes til det som faktisk finnes (Pages-URL-er, SW, localStorage). MIGRATION.md sier dette eksplisitt | – |
| D12 | **Mørkt tema alltid, Tiffany som aksent** (eiers instruks, overstyrer opprinnelig hvit bakgrunn i fase 3-spesifikasjonen) | På mørk bunn består #0ABAB5 WCAG AA også for liten tekst (~6,9:1) – tekstbegrensningen fra lys modus bortfaller. Diagram-merker bruker validert mørk-modus-trinn #0AA79F (`--saga-chart`, dataviz-validatoren: lysbånd + kontrast PASS); tekst på primærflater er mørk (#04302E) fordi hvit på Tiffany kun gir ~2,4:1. Bevisst enkelt-tema – ingen prefers-color-scheme-veksling | Ved designrevisjon |
| D13 | **SAGA OS = Control Center forfremmet til rot-skall** (rewire, ikke rebuild) | Nyeste og mest polerte UI-kode med palett/ruting/mobilnav løst og 122 sjekker i ryggen; å skrive nytt skall ville duplisert alt, å montere tre fulle sider i hverandre ville krevd iframes eller brutt `document`-basert modulkode. Styret og assistenten kom inn som flater via `window.OS.registerView` | Hvis en modul trenger full isolasjon |
| D14 | **Eget repo med separasjonsvakt i CI; ren import; publiseringssti `landing/<slug>/`** | Sideprosjekt skal være permanent atskilt fra opphavskonteksten (eiers instruks). Ren import fordi SAGA lå på fire rot-stier; vakten (`tests/separation.test.js`) håndhever navneforbudet maskinelt. Publish-stien byttet fordi `tests/` nå er testkatalog – ingen publiserte sider fantes på opphavets main, så ingen live-URL brakk (D5 respektert) | – |
| D15 | **Dokken og Assistent-flaten deler ett DOM-tre og én samtale; egen slank chat-motor i core/chat.js** | Én samtale er produktkravet (ingen splittet historikk); DOM-flytting bevarer state gratis. PWA-ens rike klient (stemme, SW) gjenbrukes ikke i skallet – bevisst duplikat, konsolidering ligger i forbedringskøen | Ved konsolidering av LLM-klienter |
| D16 | **Nattskiftet: Actions er serveren, `data/` er databasen, Issues er varslingskanalen** | Eneste serverløse vei til «jobber mens du sover» i denne stacken, uten ny driftsbyrde: cron gratis på offentlig repo, git-historikk som revisjonsspor, Issue-etikett gir mobil-push via GitHub-appen. Harde tak + kill-switch i config gjør kostnaden umulig å miste kontrollen på | Hvis stacken får ekte backend |
| D17 | **iOS bundle-ID og Keychain-tjeneste byttet til `com.jonathanfoss.saga`** | Gammel ID bar opphavsnavnet og ville brutt separasjonskontrakten. Konsekvens: reinstallasjon + ny nøkkel på iOS (dokumentert i SEPARATION.md) | – |
